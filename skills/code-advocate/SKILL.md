# Code Advocate — Adversarial Code Validation

## Purpose

Code Advocate subjects AI-generated code to an adversarial validation process before it enters the codebase. Instead of relying on a single LLM to review its own output, Code Advocate creates two opposing roles — a **Defender** who argues the code is correct, and a **Prosecutor** who tries to break it. The resulting "verdict" gives the developer a structured, evidence-based assessment.

## Core Concept

Single-model review has a fundamental weakness: LLMs tend to agree with themselves. When asked "is this code correct?", the model that generated it will often say "looks good". Adversarial validation fixes this by forcing two perspectives to clash:

- **Defender**: Constructs the best possible case for why the code is correct and safe.
- **Prosecutor**: Actively searches for flaws, edge cases, hidden assumptions, and system incompatibilities.

The tension between these roles surfaces issues that neither role would find alone.

## When to Use

- After generating code with AI, before committing
- When reviewing code that was generated by vibe coding sessions
- When unsure whether AI-generated code handles edge cases
- Before merging PRs that contain significant AI-generated portions
- When evaluating whether generated code aligns with project conventions

## Workflow

### Phase 1: Code Collection

Identify the target code to validate. This can be:

- A specific function or file the user points to
- Recently generated code in the current session
- Uncommitted changes in the working tree

Run `git diff` or read the relevant files to collect the code under review.

### Phase 2: Defense Construction (Defender Role)

Analyze the code and construct a structured defense covering:

1. **Intent**: What problem does this code solve? Is the intent clear and correct?
2. **Correctness Argument**: Why is the logic sound? Trace through the happy path.
3. **Edge Case Handling**: What edge cases are handled? How?
4. **Assumption Declaration**: What assumptions does the code make about its environment, callers, and data?
5. **Convention Compliance**: Does it follow project conventions (naming, error handling, layering)?
6. **System Compatibility**: How does it integrate with existing code? Any conflicts?

Output format for the defense — see `references/verdict-format.md` for the full template.

### Phase 3: Prosecution (Prosecutor Role)

Switch perspective. The Prosecutor's job is to find weaknesses. For each defense point, the Prosecutor:

1. **Challenges Intent**: Is the stated intent really what the code does? Look for mismatches.
2. **Constructs Counterexamples**: For each correctness argument, build specific inputs that could break it.
3. **Finds Unhandled Edge Cases**: What edge cases did the Defender miss? Consider: nil values, empty slices, concurrent access, network failures, timeout scenarios.
4. **Questions Assumptions**: For each declared assumption, ask "what if this assumption is violated?" and trace the consequences.
5. **Checks Convention Violations**: Compare against actual project conventions by reading CLAUDE.md, existing code patterns, and project structure.
6. **Identifies Integration Risks**: Look for interface mismatches, version incompatibilities, circular dependencies.

The Prosecutor must be **specific** — not "this might have a bug" but "when `userID` is an empty string, line 42 will panic because..." with file paths and line numbers.

### Phase 4: Cross-Examination

For each prosecution point, the Defender gets one chance to respond:

- **Concede**: If the prosecution is valid, acknowledge it and mark the issue as confirmed.
- **Refute**: If the prosecution is wrong, explain why with evidence (code path tracing, test coverage, type guarantees).

No second rebuttals. This keeps the process bounded.

### Phase 5: Verdict

Produce a structured verdict document. See `references/verdict-format.md` for the complete template. The verdict contains:

| Section | Content |
|---------|---------|
| **Admitted** | Points where both sides agree the code is correct |
| **Disputed** | Points where the Defender refuted the prosecution — developer should decide |
| **Conceded** | Points where the Defender accepted the prosecution — must be fixed |
| **Unverified** | Points that neither side could resolve with available information — needs testing |
| **Recommendation** | One of: PASS / CONDITIONAL PASS / REJECT |

### Phase 6: Fix Loop (Conditional)

If the verdict is CONDITIONAL PASS or REJECT:

1. Fix the conceded issues
2. Re-run the adversarial process on the fixed code (but only on the changed portions)
3. Repeat until PASS or the developer decides to accept the risk

## Important Rules

- The Prosecutor must never accept "this is unlikely" as a defense. Unlikely things happen in production.
- The Defender must never appeal to authority ("the AI generated it, so it's probably fine"). Only evidence counts.
- Both roles must reference **specific lines of code** — no vague statements.
- The verdict must always include a clear recommendation, not just a list of issues.
- If the code is trivially simple (e.g., a one-line getter), skip the full process and note why it was skipped.

## Integration with Development Flow

- **Pre-commit hook**: Run Code Advocate automatically on staged files before commit
- **PR review**: Run on the full diff when a PR is created
- **Ad-hoc**: Developer triggers manually via `/code-advocate` command
